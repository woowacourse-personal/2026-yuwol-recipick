## 1. 기술 스택 결정과 근거

### 최종 결정

**React 기반 PWA (Progressive Web App)**로 개발한다.

### 선택지 비교

**React Native (네이티브 앱)**
- 장점: 진짜 앱 경험, 클립보드/센서/알림 접근 자유
- 단점: 개발 시간 3배 이상, TestFlight 등 배포 프로세스 복잡, 인터뷰 참가자에게 앱 설치 요청 = 진입 장벽 큼
- 프로토타입 단계에서는 오버엔지니어링

**웹뷰 (WebView) 방식**
- 개념: 네이티브 앱 껍데기 안에 웹 페이지를 띄우는 방식. 겉은 앱, 속은 웹.
- 장점: 웹 개발 그대로 사용하면서 앱스토어 배포 가능
- 단점: 진짜 웹앱도 진짜 네이티브도 아닌 어중간한 결과. 성능과 UX 모두 하위. 프로토타입에는 부적합.

**PWA (Progressive Web App)** ← 선택
- 개념: 웹앱이지만 "홈 화면에 추가" 기능으로 앱처럼 실행 가능한 형태
- 장점:
  - URL만 공유하면 인터뷰 참가자가 즉시 접근 가능 (진입 장벽 최소)
  - 원하면 홈 화면에 추가하여 앱처럼 사용 가능
  - 클립보드 API, Wake Lock, 카메라 등 필요한 대부분의 기능 사용 가능
  - Vercel로 배포하면 push 한 번에 반영
- 단점: 푸시 알림은 iOS에서 제약. 다만 이번 스코프에서 알림은 사용하지 않으므로 무관.

### 실제 서비스로 전환 시

프로토타입에서 검증이 완료되면 다음 순서로 확장을 검토한다.
- 1단계: PWA를 그대로 유지하며 도메인, 사용자 계정, 백엔드 데이터베이스 추가
- 2단계: 사용자 규모가 유의미해지면 React Native로 재작성 검토
- 카드 UI 컴포넌트를 미리 프레임워크 독립적으로 설계해두면 이후 이전이 수월하다

## 2. LLM 통합 방식

### 결정

Vercel Serverless Function을 프록시로 사용하여 Anthropic Claude API를 호출한다.

### 이유

**직접 브라우저에서 호출**
- API 키가 클라이언트 코드에 노출됨 (치명적)
- CORS 이슈로 자막 추출도 브라우저에서 불가능

**Serverless Function 방식** ← 선택
- API 키는 서버 환경변수로 안전하게 관리
- CORS 우회 가능
- Vercel에 배포 시 자동 스케일링
- 로그를 통해 실사용 패턴 파악 가능 (가설 검증에 필요)

**별도 백엔드 서버**
- 프로토타입 단계에서는 오버엔지니어링
- 관리 부담이 검증 활동보다 커짐

### API 파이프라인

```
[유저] URL 입력
  ↓
[Vercel Function] /api/parse-recipe
  1. YouTube 자막 추출 (youtube-transcript 계열)
  2. Claude API 호출 with Tool Use (구조화된 JSON 강제)
  3. 결과 반환
  ↓
[클라이언트] 파싱 결과 localStorage에 저장
```

### Claude API 활용 시 핵심 포인트

Claude API의 **Tool Use** 기능을 사용하면 응답을 정확한 JSON 스키마로 강제할 수 있다. 이는 파싱 오류를 최소화하는 검증된 방식이다. 프롬프트에서 "이 도구를 사용해서 응답해달라"고 명시하고, tool_use 응답만 파싱한다. 별도의 JSON 파싱 로직이나 재시도 로직 없이도 안정적이다.

## 3. YouTube 임베드 실험

### 결정

이번 v5에서는 임베디드를 반드시 구현하고 검증한다.

### 배경

Gemini Canvas에서 임베드가 동작하지 않았던 이유는 Gemini의 iframe 정책 제약 때문일 가능성이 높다. 자체 배포한 웹앱에서는 YouTube IFrame Player API를 통해 완전한 임베드가 가능하다.

### 구현 방식

**YouTube IFrame Player API** 사용
- 특정 타임스탬프에서 시작 가능
- 재생/일시정지/구간 반복 등 제어 가능
- 크리에이터가 임베드를 차단한 경우에만 예외 처리 필요

### 검증 목적

v4 파라디의 강력한 요청("임베드로 넣으면 완벽할 것 같다")이 실제 사용 시에도 유효한지 확인. 또한 재현 도구 포지셔닝에서 임베드가 필요한지에 대한 v3의 결정을 재검증한다.

## 4. 지금까지의 학습 요약

### 프로덕트 개요

레시픽은 유튜브 레시피 URL을 입력하면 LLM이 자막을 파싱하여 단계별 카드로 변환하고, 저장 시점부터 요리용 포맷으로 아카이빙하는 서비스이다.

### 인터뷰 12명의 핵심 발견

**검증된 것**
- 스텝 단위 카드 분할과 하이라이팅은 요리 도중 참고 관점에서 유효
- 저장 시스템 부재는 실제 페인포인트 (참가자들이 캡처, 종이 메모, 블로그 등 자체 시스템 운영 중)
- 아카이빙과 카테고리 요구는 반복 확인됨

**흔들리거나 미검증**
- 손 물기 페인포인트: 12명 중 5명만 명확한 페인. 신뢰도 하락.
- 재현 도구 포지셔닝: v3 방향 결정 후 v4에서 검증되지 않음. 유즈케이스 정의가 12명 중 8명이 모두 다름.
- 실사용 관찰 여전히 미실행. 모든 반응이 상상 기반.

**근본적 재고 필요**
- 카드 UI 자체: 무빙(v3), 파라디(v4)가 독립적으로 "전체 뷰 우선 + 카드는 상세 진입" 구조 제안
- 임베디드 플레이어: v3에서 배제했으나 파라디(v4)가 강력한 논리로 재요청
- 접근성 마찰: 4회에 걸쳐 지적됨. 클립보드 감지만으로는 부족.

**새로 부상한 방향**
- 파라디(v4)의 유연 검색 강화 방향. "면 요리" 검색 시 파스타면, 우동면 등 관련 재료 모두 매칭. 재방문 트리거 확보 후보.

### 반복 요청되었으나 스코프 외로 유지되는 것

공유 기능, 여러 레시피 비교 분석, 음성 명령, 알림, AI 질의응답, 재료 기반 추천, 커머스 연동. 이번 스코프에서도 배제한다.

## 5. v5 프로토타입의 검증 목적

이번 v5는 단순히 기능을 옮겨오는 것이 아니라 다음 네 가지를 실사용 관찰로 검증한다.

**검증 1: 카드 UI vs 전체 뷰 우선**
파라디의 근본적 제안을 검증한다. 두 가지 뷰 모드를 모두 제공하여 유저가 실제로 어느 쪽을 선택하는지 관찰.

**검증 2: 임베디드 플레이어**
실제로 임베디드가 요리 상황에서 유효한지, 아니면 여전히 유튜브 이동이 낫다고 느끼는지 확인.

**검증 3: 아카이브 재접근 패턴**
저장한 레시피의 30일 재조회율 (가설 H2). 실사용 로그로 확인.

**검증 4: 접근성 마찰**
클립보드 자동 감지, 검색 강화가 실제 재방문 트리거로 작동하는지 확인.

```typescript
type Recipe = {
```

## 8. 화면 구조

### 화면 1: 홈 (아카이브 중심)

**상단**
- URL 입력창 + "저장" 버튼
- 클립보드 자동 감지: 페이지 진입 시 유튜브 URL이 클립보드에 있으면 토스트로 "이 레시피를 저장할까요?" 표시. 확인 시 자동 파싱 진행. 같은 URL은 재감지하지 않음.

**중단: 검색**
- 검색창 (제목, 재료, 태그 통합 검색)
- 파라디 요청 반영: "면"으로 검색하면 재료에 "파스타면", "우동면", "소면" 등이 있는 레시피 모두 매칭. 이를 위해 Claude 파싱 시 태그를 함께 생성.
- 태그 예시: "면 요리", "매운맛", "국물 요리", "한식" 등

**중단: 필터**
- 사용자 커스텀 카테고리 탭
- 매체 필터: 전체 / 영상 참고 / 직접 작성 (찰리 요청)
- 정렬: 최근 조회순, 최근 저장순, 자주 조회순

**하단: 레시피 목록**
- 카드 형태로 표시
- 각 카드: 썸네일, 제목, 채널명(출처), 저장일, 조회 횟수

### 화면 2: 저장 진행

- URL 파싱과 Claude 호출 중 로딩
- 성공 시 카테고리 선택 모달 (기존 카테고리 또는 새로 추가)
- 실패 시 명확한 에러

### 화면 3: 재료 준비

**상단**
- 레시피 제목
- 크리에이터 출처: "백종원의 요리비책 영상을 참고했어요" (탭 시 유튜브 이동)
- 수정 버튼 (우측 상단)

**중단: 재료 체크리스트**
- 파라디 요청 반영: 기본 재료(물, 소금, 후추 등)와 특수 재료를 섹션으로 분리
- 기본 재료는 접었다 펼 수 있으며 기본은 접힘

**하단: 요리 순서 미리보기**
- 전체 스텝을 축약 형태로 한눈에 표시 (무빙 요청)
- 파라디 지적 반영: 이 미리보기가 화면의 주인공이 되도록 배치 (재료 아래가 아니라 재료와 대등하게)
- 각 스텝 탭 시 쿠킹 모드의 해당 스텝으로 이동 가능

**최하단**
- "요리 시작" 버튼 (뷰 모드 선택 가능)
  - 카드 모드 (스텝별 카드)
  - 전체 모드 (스크롤 뷰)

### 화면 4: 쿠킹 모드 (핵심 실험 지점)

**두 가지 뷰 모드를 모두 지원한다. 유저가 상단 토글로 전환할 수 있다.**

#### 카드 모드 (기존 방식)

- 상단: 레시피 제목, 현재 스텝 번호 (예: "3 / 7")
- **YouTube 임베디드 플레이어** (카드 상단)
  - 해당 스텝의 타임스탬프에서 시작
  - 스텝 이동 시 자동으로 seek
  - 크기는 화면 상단 3분의 1 정도 (파라디: "화면 작아져도 괜찮다")
- 카드 중앙: 스텝 텍스트, 하이라이팅, 유저 메모 (있을 시)
- 좌우 스와이프 + 이전/다음 버튼
- 하단: 프로그레스 바
- 전체 보기 아이콘: 탭 시 모달로 전체 스텝 표시

#### 전체 모드 (파라디 제안)

- 전체 레시피를 위아래로 스크롤 가능한 형태로 표시
- 각 스텝이 세로로 나열
- 유튜브 임베디드 플레이어는 상단 고정 (스크롤해도 따라옴)
- 임베디드 플레이어에서 현재 재생 중인 시점의 스텝을 자동 하이라이트
- 각 스텝의 오른쪽에 "상세 보기" 버튼 (탭 시 카드 뷰로 전환)
- 조작 최소화 (파라디 핵심 논리: "조작이 문제의 핵심이라면 이게 낫다")

**뷰 모드 토글의 위치와 로깅**
- 상단 우측에 뚜렷하게 배치
- 유저가 전환할 때마다 `toggleView` 이벤트 로깅
- 이후 재조회 시 마지막 사용한 뷰 모드가 기본값
- 이 로그가 검증 1의 핵심 데이터

**화면 꺼짐 방지**
- 쿠킹 모드 진입 시 Wake Lock API 활성화
- 화면 이탈 시 해제

### 화면 5: 수정 모드

- 재료 준비 화면에서 진입
- 인라인 수정 (별도 화면 전환 없음)
- 수정 가능: 제목, 카테고리, 재료 목록, 스텝 텍스트, 스텝별 메모
- 스텝별 메모는 파도(v3) 요청. 요리 후 본인이 변형한 부분 기록.

### 화면 6: 직접 작성

- 레시피 제목 (필수)
- 원본 링크 (선택)
- 카테고리 선택
- 재료 (기본/특수 구분 가능, 계량 입력)
- 조리 스텝 (동적 추가, 각 스텝에 메모 필드)
- 저장 시 아카이브에 즉시 반영

## 9. Claude API 파싱 프롬프트

Serverless Function에서 Claude API 호출 시 다음 시스템 프롬프트를 사용한다.

```
당신은 요리 유튜브 영상의 자막을 분석하여 구조화된 레시피로 변환하는 전문가입니다.

입력된 자막을 분석하여 다음을 추출하세요:

1. 레시피 제목: 영상 제목에서 실제 요리명만 추출
2. 재료 목록:
   - 각 재료의 이름과 계량
   - 기본 재료(물, 소금, 후추, 식용유 등)와 특수 재료 구분
3. 조리 스텝:
   - 각 스텝은 "재료를 가지고 무엇을 한다" 형태로 원자적으로 작성
   - 여러 동작이 하나의 스텝에 섞이지 않도록 분리
   - 각 스텝에 자막상의 타임스탬프 매핑
   - 각 스텝의 1줄 요약 별도 제공 (오버뷰용)
   - 재료명, 계량, 시간, 화력 정보를 하이라이트 대상으로 분류
4. 태그: 이 요리의 특성을 나타내는 태그 (예: "면 요리", "국물 요리", "매운맛", "한식")
   - 검색 시 관련 요리를 함께 노출하기 위한 것
   - 3~7개 정도

structure_recipe 도구를 사용하여 결과를 반환하세요.
```

Tool Use 스키마를 통해 JSON 형식을 강제하고, 클라이언트에서는 tool_use 블록만 파싱한다.

## 10. 개발 우선순위

## 11. 디자인 원칙

1. **요리 상황을 우선한다**: 젖은 손, 팔 길이 시야 전제. 폰트와 터치 영역 크게.
2. **강제하지 않는다**: 자동 넘김, 강제 대기 없음. 유저 페이스 존중.
3. **스텝은 원자적이다**: 카드 하나에 한 동작. "재료를 가지고 무엇을 한다"로 종결.
4. **전체 파악이 가능해야 한다**: 요리 시작 전 전체 흐름 파악을 지원.
5. **출처를 명시한다**: 크리에이터 채널명을 항상 접근 가능하게.
6. **두 뷰를 대등하게 취급한다**: 카드와 전체 뷰 중 어느 것도 우위로 두지 않는다. 유저 선택을 로깅으로 관찰한다.

## 12. 예외 처리

- **유튜브 자막 없음**: "이 영상은 자막이 없어 저장할 수 없습니다"
- **블로그/릴스 URL**: "현재는 유튜브만 지원합니다"
- **Claude 파싱 오류**: Tool Use 사용으로 오류율은 낮으나, 실패 시 1회 재시도 후 실패 처리
- **동일 URL**: 캐싱된 결과 재사용 (API 비용 절감)
- **임베드 차단된 영상**: 임베디드 대신 "유튜브에서 보기" 버튼으로 폴백
- **클립보드 권한 거부**: 무시하고 수동 입력으로 진행

## 15. 성공 판정 기준

1. **파이프라인 안정성**: 자막 있는 요리 영상 80% 이상에서 사용 가능한 결과
2. **실사용 확보**: 최소 3명이 최소 4주간 실사용, 로그 수집 완료
3. **뷰 모드 검증**: 카드 뷰와 전체 뷰 중 어느 것이 선호되는지 데이터로 확인
4. **임베디드 검증**: 임베디드가 실제 요리 상황에서 사용되는지, 아니면 무시되는지 확인
5. **접근성 개선 확인**: 클립보드 감지가 실제 진입점으로 사용되는지, 검색이 재방문 트리거로 작동하는지 확인

## 16. 개발 시 주의사항

- **저작권**: 크리에이터 attribution(채널명, 원본 URL)은 반드시 유지. 저작권 관련 문구는 하단이나 설정에 명시.
- **API 비용**: 동일 URL 캐싱 필수. 자막이 긴 영상은 토큰 소비가 크므로 로컬 캐싱과 서버 캐싱 양쪽 구현.
- **모바일 우선**: 모든 UI를 모바일 기준으로. 데스크탑은 반응형으로만 대응.
- **로깅은 필수**: 이 프로토타입의 존재 이유는 가설 검증. 로깅 없이 배포하면 검증 자체가 불가능.
