## 핵심 발견
- **검증된 것 / 강한 신호**
  - **카드-타임라인 연동 = 킬러 기능.** 왓슨 "큰 차별점", 이안 "기존 불편 해소될 것", 송송 긍정(단 안정성), 제제 간접 긍정. **4명이 명시 언급.** → `H-EMBED`의 강한 당김. 단 "될 때도 있고 안 될 때도" = **안정성이 검증의 관문**.
  - **타이머 긍정**(왓슨: 다음 카드 갔다 와도 유지되는 점 특히 굿) — 세션 매니저 설계가 유효. 단 개인 스타일 의존은 여전([[2026-07-21-siyeon-first-realuse]] 시현은 불필요).
- **가장 큰 실제 마찰 (5명)**
  - **파싱 정확도.** 왓슨(재료/순서 누락, **자막 파싱 < 댓글/설명란 직접 붙여넣기** 퀄리티), 이안(영상만의 정보 누락), 제제(AI가 너무 세밀하게 쪼갬), 송송(중요 과정 누락), 벤지(텍스트만으론 부족). → 실사용 시현의 오파싱과 합쳐 **프로덕트 최소 조건**으로 확정.
  - **인분·계량 정보 필수 (4명).** 벤지("몇 인분 기준 + 인분 조정"), 밍구(양·인분 미표시), 제제(양 미표시), 송송(큰술 등 기준 부재). → 파싱 스키마에 `servings` 필수 편입 문제.
- **근본적 재고 필요**
  - **`H-RETENTION` — 아카이빙 니즈 양극화.** 재현 긍정: 제제·송송·밍구·벤지. 즉시성 부정: 왓슨("필요할 때 찾음, 새 레시피 발견 선호"), 이안(유튜브 좋아요로 충분). **저장 습관 있어도 앱형 아카이빙 니즈는 못 정의** — 밍구 1000개 저장하고도 안 찾음. **해석: 니즈는 아카이빙 자체가 아니라 검색·발견. 저장은 수단.** 송송 강력 긍정: "그 레시피 뭐였더라 페인 해결, 탐색 시간 단축."
  - **`H-VIEW` — 다음 스텝 미리보기 요구 vs 오버뷰 거부.** 왓슨: "요리하며 다음 단계가 궁금 → 한 카드에 다음 스텝 축약을 같이." 그러나 목록(오버뷰)은 **너무 요약돼 안 봄** → "차라리 다음을 눌러 갔다옴." 카드 한 장=한 스텝 원칙과 충돌. **오버뷰가 답이 아닐 수 있다는 신호**(시현의 오버뷰 실패와 일치).
  - **홈 정보구조 재설계 (3명).** 왓슨(아카이빙 탭 불필요, 홈 스크롤로 충분), 이안(최근≒아카이빙 차이 없음, **배민식 카테고리 우선**), 송송(아카이빙 필터링 기대했으나 미동작).
- **`H-ACCESS` — URL 진입 마찰 근본 해결 요구 (4명).** 왓슨·밍구(폴센트 레퍼런스)의 **아이폰 공유시트 통합**, 송송(앱 내 유튜브 검색), 벤지(클립보드 자동 붙여넣기). 공유시트는 네이티브 API 필요 → **RN 전환 근거로 축적**(PWA 한계).
