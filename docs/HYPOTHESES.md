> 이 문서가 리서치 하네스의 **척추**다. "우리는 무엇을 검증 중인가"의 단일 진실 소스.
> 인터뷰·실험 결과가 들어오면 **먼저 여기 해당 가설의 상태·근거를 갱신**하고,
> 상세 데이터는 `research/`·`experiments/`의 개별 파일에 남긴 뒤 `[[링크]]`로 연결한다.
>
> - **상태 값**: `미검증` · `지지`(evidence 쌓임) · `흔들림`(혼재/약함) · `기각` · `보류`
> - PRD §5의 4대 검증 목적을 가설로 구조화한 것. 원본 근거는 `PRD-v5.md` §4~5.
> - 새 가설이 생기면 아래 표에 행 추가 + 상세 블록 추가. ID는 의미 기반(semantic)으로 부여.

## ⚠️ 상류 리스크 — 파싱 정확도 (모든 가설의 전제 조건)

- **2026-07-22 부상**: 첫 실사용(시현)에서 재료 7개 중 일부 누락 + **"굴소스→불소스" 오파싱**. 시연 6명 중 5명도 누락·과분할 지적. 왓슨: **자막 파싱 < 설명란/댓글 직접 붙여넣기** 퀄리티.
- **함의**: 파싱이 틀리면 어떤 가설도 공정하게 검증되지 않는다. **개선 전 재실험은 정보가치가 낮음.** 가설이 아니므로 여기선 리스크로만 기록 — 대응은 파싱 소스 확장(자막+설명란+상위 댓글)·프롬프트 재설계·결과 검증 로직 + 스키마 `servings` 필수화. [[2026-07-21-siyeon-first-realuse]] [[2026-07-22-interviews-v5-6people]]

## H-VIEW — 카드 뷰 vs 전체 뷰

- **진술**: 유저는 카드 단위 뷰보다 전체 뷰(스크롤 개요)를 기본으로 선택한다. (v3 무빙·v4 파라디가 독립적으로 제안)
- **측정**: 쿠킹 진입 후 `toggleView` 로그의 방향·빈도, 각 뷰 체류 비율.
- **판정 기준(초안)**: 관찰 세션의 과반이 전체 뷰에서 요리를 마치면 `지지`. 혼재면 `흔들림`.
- **현재 상태**: `미검증` — 실사용 첫 관찰에서 **오버뷰 구현이 니즈를 못 채움**(역방향 신호).
- **근거**: 시현 실사용 — "재료 준비 화면 요리순서가 추상적, 직관성 부족"(오버뷰 실패). 왓슨 — "다음 스텝은 궁금하나 목록(오버뷰)은 너무 요약돼 안 봄, 차라리 다음을 눌러 갔다옴". [[2026-07-21-siyeon-first-realuse]] [[2026-07-22-interviews-v5-6people]]
- **함의/다음 액션**: 전체 뷰 우선 가설이 흔들릴 여지 — 유저가 원하는 건 "전체 오버뷰"가 아니라 **카드 내 다음 스텝 미리보기(축약)**일 수 있음. 두 뷰는 대등 유지하되 다음 스텝 미리보기를 별도 검증.

## H-EMBED — 임베디드 플레이어 유효성

- **진술**: 요리 중 임베디드 플레이어가 유튜브 앱 이동보다 유효하다. (v3에서 배제 → v4 파라디가 재요청)
- **측정**: `openEmbed` 발생률, 특정 스텝 seek 사용, 임베드 vs 외부 이동 선택.
- **판정 기준(초안)**: 관찰 세션에서 임베드 실사용이 반복되면 `지지`.
- **현재 상태**: `미검증` — 시연에서 **강한 당김**(카드-타임라인 연동이 "큰 차별점"으로 반복 지목), 그러나 실사용 미관찰 + 현재 동작 불안정.
- **근거**: 6인 중 4명이 카드-타임라인 연동 명시 언급(왓슨 "큰 차별점", 이안·송송·제제). "될 때도 있고 안 될 때도" = 안정성 부족. [[2026-07-22-interviews-v5-6people]]
- **함의/다음 액션**: **안정성 100%가 검증의 관문.** 다음 카드 이동 시 영상 자동 정지(송송 요청 — 다음 단계 준비 시간). 스텝별 seek 타임스탬프 로깅 정확도 확인.

## H-RETENTION — 아카이브 재접근

- **진술**: 저장 시스템 부재가 실제 페인이며, 저장 기능은 30일 내 유의미한 재조회를 만든다.
- **측정**: `view` 재접근 간격, `accessCount` 분포, 저장 후 N일 재방문율.
- **판정 기준(초안)**: 저장 레시피의 30일 재조회율이 목표선(추후 설정) 이상이면 `지지`.
- **현재 상태**: `흔들림` — 저장 페인 자체는 반복 확인되나, **저장 습관이 있어도 앱형 아카이빙 니즈는 못 정의**. 밍구 1000개 저장하고도 안 찾음(저장≠재조회).
- **근거**: 인터뷰 12명 자체 저장 시스템 운영 [[2026-07-18-interviews]]. 6인 배치 — 재현 긍정(제제·송송·밍구·벤지) vs 즉시성 부정(왓슨·이안). **진짜 니즈는 아카이빙 자체가 아니라 검색·발견, 저장은 수단.** [[2026-07-22-interviews-v5-6people]]
- **함의/다음 액션**: 아카이빙을 **검색·발견 중심으로 재포지셔닝** → H-ACCESS와 수렴. 목표 재조회율 수치 확정 필요.

## H-ACCESS — 접근성 마찰 / 재방문 트리거

- **진술**: 클립보드 자동 감지 + 유연 검색("면 요리"→파스타면·우동면 매칭)이 재방문 트리거로 작동한다.
- **측정**: 클립보드 감지→save 전환율, 검색 사용 후 재방문, 검색어 다양성.
- **판정 기준(초안)**: 검색·클립보드 경유 재방문이 반복 관찰되면 `지지`.
- **현재 상태**: `미검증` — 접근성 마찰은 인터뷰 4회 지적됐으나 클립보드만으로는 부족 지적됨.
- **근거**: 인터뷰 접근성 마찰 4회 지적. [[2026-07-18-interviews]]
- **함의/다음 액션**: 유연 검색이 실제 트리거인지 별도 실험 설계.
- **2026-07-20 갱신**: 저장 진입 마찰을 넓히는 방향으로 **텍스트 붙여넣기 파싱** 경로 추가 — 자막 없는 영상(설명란·고정 댓글 글)과 블로그/손메모까지 담을 수 있게 함. `save` 로그의 `metadata.sourceType`(`youtube`/`manual`/`text`)로 어느 경로가 실제 저장을 유발하는지 관찰. [[INSIGHTS-LOG]]
- **2026-07-22 갱신**: 진입 마찰이 시연 6명 중 4명 + 실사용(시현)에서 재확인. **공유시트 통합**(왓슨·밍구, 폴센트 레퍼런스)이 근본 해결로 반복 지목 → 네이티브 API 필요, **RN 전환 근거로 축적**. 클립보드 자동 붙여넣기(벤지)·앱 내 유튜브 검색(송송)도 요청. H-RETENTION의 "검색·발견" 니즈가 이 가설로 수렴. [[2026-07-21-siyeon-first-realuse]] [[2026-07-22-interviews-v5-6people]]
