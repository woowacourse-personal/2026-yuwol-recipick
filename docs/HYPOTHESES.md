## H-VIEW — 카드 뷰 vs 전체 뷰

- **진술**: 유저는 카드 단위 뷰보다 전체 뷰(스크롤 개요)를 기본으로 선택한다. (v3 무빙·v4 파라디가 독립적으로 제안)
- **측정**: 쿠킹 진입 후 `toggleView` 로그의 방향·빈도, 각 뷰 체류 비율.
- **판정 기준(초안)**: 관찰 세션의 과반이 전체 뷰에서 요리를 마치면 `지지`. 혼재면 `흔들림`.
- **현재 상태**: `미검증` — 실사용 관찰 미실행.
- **근거**: (없음)
- **함의/다음 액션**: 두 뷰를 대등하게 유지. 어느 쪽도 기본 우위 두지 말 것 (관찰 오염 방지).

## H-EMBED — 임베디드 플레이어 유효성

- **진술**: 요리 중 임베디드 플레이어가 유튜브 앱 이동보다 유효하다. (v3에서 배제 → v4 파라디가 재요청)
- **측정**: `openEmbed` 발생률, 특정 스텝 seek 사용, 임베드 vs 외부 이동 선택.
- **판정 기준(초안)**: 관찰 세션에서 임베드 실사용이 반복되면 `지지`.
- **현재 상태**: `미검증`.
- **근거**: (없음)
- **함의/다음 액션**: 스텝별 seek 타임스탬프 로깅 정확도 확인.

## H-RETENTION — 아카이브 재접근

- **진술**: 저장 시스템 부재가 실제 페인이며, 저장 기능은 30일 내 유의미한 재조회를 만든다.
- **측정**: `view` 재접근 간격, `accessCount` 분포, 저장 후 N일 재방문율.
- **판정 기준(초안)**: 저장 레시피의 30일 재조회율이 목표선(추후 설정) 이상이면 `지지`.
- **현재 상태**: `미검증` (인터뷰선 저장 페인 자체는 반복 확인됨 → PRD §4 "검증된 것").
- **근거**: 인터뷰 12명 — 참가자들이 캡처/종이/블로그로 자체 저장 시스템 운영 중. [[2026-07-18-interviews]]
- **함의/다음 액션**: 목표 재조회율 수치 확정 필요.

## H-ACCESS — 접근성 마찰 / 재방문 트리거

- **진술**: 클립보드 자동 감지 + 유연 검색("면 요리"→파스타면·우동면 매칭)이 재방문 트리거로 작동한다.
- **측정**: 클립보드 감지→save 전환율, 검색 사용 후 재방문, 검색어 다양성.
- **판정 기준(초안)**: 검색·클립보드 경유 재방문이 반복 관찰되면 `지지`.
- **현재 상태**: `미검증` — 접근성 마찰은 인터뷰 4회 지적됐으나 클립보드만으로는 부족 지적됨.
- **근거**: 인터뷰 접근성 마찰 4회 지적. [[2026-07-18-interviews]]
- **함의/다음 액션**: 유연 검색이 실제 트리거인지 별도 실험 설계.
- **2026-07-20 갱신**: 저장 진입 마찰을 넓히는 방향으로 **텍스트 붙여넣기 파싱** 경로 추가 — 자막 없는 영상(설명란·고정 댓글 글)과 블로그/손메모까지 담을 수 있게 함. `save` 로그의 `metadata.sourceType`(`youtube`/`manual`/`text`)로 어느 경로가 실제 저장을 유발하는지 관찰. [[INSIGHTS-LOG]]
