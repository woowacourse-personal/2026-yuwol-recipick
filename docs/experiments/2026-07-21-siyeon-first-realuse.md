# 첫 실사용 관찰 — 시현 (2026-07-21)

> **하네스가 v1부터 12명 내내 "미실행"으로 남겨둔 실사용 관찰의 첫 데이터.**
> 지금까지 모든 반응은 상상 기반이었고, 이 한 건이 상상 응답 12명만큼의 정보를 줬다.
> 상세 원문·AI 정리본은 마스터 인터뷰 문서 `260721_v5` 참조.

## 검증 대상 가설
- 개선 가설(비공식): "유튜브 원본과 레시픽을 동시에 쓸 수 있을 때 레시픽을 더 자주 참고할 것" → **명확한 실패**. 시현은 유튜브 쇼츠를 선택, 레시픽을 요리 중 참고 안 함.
- 부수적으로 `H-ACCESS`(진입 마찰), `H-VIEW`(전체 흐름 파악), 그리고 **모든 가설의 상류 조건인 파싱 정확도**를 건드림.

## 세팅
- 대상: 시현 — 만든 요리 **차돌 숙주 볶음**. 평소 유튜브 쇼츠로 레시피 참고.
- **주의(실험 조건 불명확)**: 인터뷰 정리만으로는 유튜브와 레시픽을 **동시 제공**했는지 **자율 선택**이었는지 불분명. 다음 실사용 설계 시 조건을 사전 고정할 것.
- **타겟 적합성 경고**: 시현은 요리 경력이 쌓여 정석 레시피를 잘 안 따르는 유저. 본인이 "초보자 기준으로는 잘 쓸 듯"이라 명시 → 타겟 밖 케이스. 한 건으로 방향을 뒤집을 근거는 아님.

## 핵심 발견
- **검증된 것 (실사용에서 처음 확정)**
  - **손 조작 페인은 실재한다.** "레시피 보려고 폰 조작할 때 손에 물·양념 묻은 경우 불편." 상상이 아닌 실제 상황의 확인. (단, 아래 "함정" 참조)
  - **링크 붙여넣기 마찰도 실재.** "복사 붙여넣기 과정이 수고스러운 것은 맞다." → `H-ACCESS` 마찰이 실사용에서도 유효.
- **근본적 재고 필요**
  - **파싱 정확도 = 프로덕트 최소 조건.** 재료 7개 중 일부 인식 실패 + **"굴소스 → 불소스" 오타**. 잘못된 재료명은 실제 요리 위험이며 신뢰도를 붕괴시킴. 상상 응답 12명에선 한 번도 안 나온, 실사용에서만 드러난 문제. **이게 안 되면 다른 모든 가설 검증이 무의미.**
  - **카드 UI가 손 조작 페인을 해결하지 못한다(함정).** 카드로 바꿔도 스와이프·버튼을 손으로 눌러야 함 → 유튜브와 동일한 조작 필요. 시현 결론: "음성 인식으로 조작 자체를 없애야 진짜 해결." 프로덕트의 차별점 명제에 대한 직접 도전.
  - **`H-VIEW` — 요리순서 미리보기(오버뷰) 구현 실패.** "재료 준비 화면의 요리 순서가 추상적이고 직관적으로 알아보기 어렵다." 무빙(v3)·파라디(v4)가 요청한 니즈 자체는 유효하나, 텍스트 요약만으론 "전체 흐름 파악" 목적을 못 채움.
- **새로 부상한 방향**
  - **냉털**: 냉장고 재료 입력 → 메뉴 추천 → 레시피 담기. 링크 붙여넣기 마찰을 완전히 우회하는 진입점. 단 스코프 확장(PRD §14 "재료 기반 추천"과 충돌 여지) → **대안 방향으로만 기록**, 즉시 도입 아님.
  - **완료 피드백 / 달력 스탬프**: "끝났을 때 완료됐다는 피드백 필요", "언제 요리했는지 인증(달력 스탬프)." 리텐션 요소지만 습관 추적기로의 스코프 확장 주의.

## 인용/원자료
> "요리할 때 레시피를 보기 위해 폰을 조작하는 상황에서 불편함을 많이 느낌. 손에 물이나 양념 등이 묻어 있는 경우."
> "편한 거는 이 프로그램이 맞다. (그러나) 손의 물기 때문에 조작해야 하는 불편은 똑같다."
> "아카이빙이라는 단어를 일반인들은 잘 모를 수 있다." (UX 카피 검토 필요)

## 결과
- **도구 선택 결과**: 유튜브 쇼츠 선택(레시픽 미사용). 추정 요인 — 파싱 오류로 신뢰도 하락 + UI가 편해 보이지 않음 + 붙여넣기 마찰 + 본인이 평소 레시피를 잘 안 봄.
- **타이머**: 본인 스타일상 불필요(감으로 시간 잼) — v2 지오와 동일 패턴. 타이머는 옵션이라는 기존 판단 재확인.
